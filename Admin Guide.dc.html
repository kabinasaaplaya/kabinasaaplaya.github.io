Running the Kabina sa Aplaya website

Everything is free: hosting on GitHub Pages, availability synced from Airbnb/Booking.com iCal, booking requests delivered straight to your Gmail inbox. No website builder, no subscription, no third-party dashboard. You edit plain files; GitHub publishes them.

⚠ One-time activation needed: the Book Now form sends requests to kabinasaaplaya@gmail.com through FormSubmit.co (free, no account). The very first time someone presses “Send booking request” on the live site, FormSubmit emails you an activation link, open that email and click Activate once. Every request after that lands in your inbox automatically as a neat table. If requests ever stop arriving, check your Spam folder and mark FormSubmit as “Not spam”.

1 · Launch on GitHub Pages (one-time, ~20 min, free forever)

Step 1. Go to github.com → Sign up (free) using kabinasaaplaya@gmail.com so the whole team can share the login.

Step 2. Click the + (top right) → New repository. Name it exactly: kabinasaaplaya.github.io, wait, use your username: if your username is kabinasaaplaya, the repo name kabinasaaplaya.github.io makes your site live at https://kabinasaaplaya.github.io. Set it to Public, then Create.

Step 3. Download this whole project as a ZIP (ask Claude: “give me the site as a download”), unzip it on your computer.

Step 4. In your new repo, click uploading an existing file (or Add file → Upload files), drag ALL the site files in (including the assets folder), then click Commit changes.

Step 5. Repo → Settings → Pages. Under “Build and deployment”, Source = Deploy from a branch, Branch = main, folder = / (root). Save. In 1–2 minutes your site is live at https://<username>.github.io.

Step 6. Already done for you: in this download the homepage is named index.html and every link points to it — nothing to rename.

Why GitHub Pages over Framer free: no “Made in Framer” badge, no page limits, you keep full control of the HTML (which is what makes the free SEO below possible), and editing availability is just editing one small file, no builder needed. Later, a custom domain (₱600–900/yr from Namecheap/Porkbun) plugs straight in under Settings → Pages → Custom domain.

2 · Update availability by hand (2 minutes, from any phone)

All booked dates live in one small file: availability.json. The calendars on the three cabin pages and the Book Now checker all read from it.

To block dates: open the repo on github.com → click availability.json → pencil icon → add a line like this inside the right cabin's list → Commit changes. Live in ~1 minute.

"makiling": [
  { "start": "2026-08-15", "end": "2026-08-16", "note": "Dela Cruz family" }
]
  • start = check-in date, end = check-out date (the check-out day itself stays FREE, same rule as Airbnb).
  • A one-night stay on Aug 15: start 2026-08-15, end 2026-08-16. A day tour blocks one day: start = that day, end = the next day.
  • Dates are always YYYY-MM-DD. Keep the commas between entries. note is only for you, it never shows on the site.
  • To free dates up, delete that line and commit.

3 · Automatic sync from Airbnb & Booking.com (optional, one-time setup)

Since you keep Airbnb/Booking.com accurate, the site can copy them automatically every 6 hours using a free GitHub Action. Once set up, you never touch availability.json again.

Get your iCal links (each cabin has one):

  • Airbnb: Host dashboard → Calendar → pick the listing → Availability settings → Connect to another website / Export calendar → copy the link ending in .ics.
  • Booking.com: Extranet → Rates & Availability → Sync calendars → Export calendar → copy the .ics link.

Install the sync: the file github-action-ical-sync.yml is already in this project. In your repo: Add file → Create new file, name it exactly .github/workflows/ical-sync.yml, paste that file's contents, replace the placeholder URLs at the top with your real iCal links, and commit. Check it works under the repo's Actions tab (you can also press “Run workflow” to test immediately).

Tip: also import each platform's iCal into the other (Airbnb ⇆ Booking.com sync) so you never get a double booking between the two, same Export/Import screens.

4 · Adding your photos

Photos are ordinary image files kept in one folder: assets/photos/. Each empty photo box on the site shows the exact filename it is waiting for. Give your image that name, put it in that folder, and it appears, on every page that uses it (a cabin's hero photo shows on the Home card, The Kabinas card, and the cabin page all at once, because they share one file).

To add or change a photo (on GitHub, after launch): open your repo, then the assets/photos folder, then Add file → Upload files, drag your image in named to match the box (for example makiling-hero.jpg), then Commit changes. To replace a photo, upload a new file with the same name. To remove one, delete the file. Changes are live in about a minute.

Filenames must match exactly (all lowercase, no spaces), for example sembrano-hero.jpg, makiling-tour-bedroom-1.jpg, am-pool.jpg. Open a page in this workspace to read the filename printed inside each empty box. .jpg, .png, and .webp all work.

Faster loading (optional but recommended): install the manifest helper so pages never waste requests looking for photos. In your repo: Add file → Create new file, name it exactly .github/workflows/photo-manifest.yml, paste the contents of github-action-photo-manifest.yml (in this project), and commit. From then on the site keeps an index of your photos automatically each time you add one.

Photographer credits (for the Events photoshoot carousel): open assets/photos/credits.json and add a line like "events-shoot-1": "Juan Cruz". The name shows on that photo.

Best sources: your existing Airbnb/Booking.com listing photos. Use landscape shots, ideally 1600px+ wide for the hero. Compress them first at squoosh.app (free, no signup) so pages load fast, big photos are the #1 thing that slows a resort site on mobile data.

5 · Getting found on Google (all free)

Already built into the site: every page has a keyword-rich title and description (“resort in Jala-jala, Rizal”, “A-frame cabin”, “Laguna de Bay”), proper heading structure, descriptive link text, mobile-first layout, and a sitemap.xml + robots.txt.

Do these once after launch:

  1. Google Search Console (search.google.com/search-console): add your site URL, verify, and submit sitemap.xml. This is THE step that gets Google to index you.
  2. Google Business Profile (business.google.com): claim “Kabina sa Aplaya – RJFLn Resort” with your Jala-jala address, photos, and the site URL. This is what puts you on Google Maps and the local results when people search “resort in Rizal”, for a local resort it matters even more than the website ranking.
  3. Link the site everywhere: Facebook page (website field + pinned post), Instagram bio, Airbnb/Booking host profile. Links from real pages tell Google you're legit.
  4. Ask happy guests for Google reviews, reviews are the strongest free local-SEO signal there is.

Expect Google to take 1–4 weeks to start showing the site. Search Console → “Performance” shows you which searches bring people in.

6 · Changing prices, rules, or text later

Every page is a plain .html file you can open in the GitHub editor (pencil icon). Prices appear on: The Kabinas page, each cabin page, and inside Book Now.dc.html (search for “5500”). Rules live on the About and Amenities pages. Use the browser's find (Ctrl+F) inside the editor to jump to the exact text, change it, commit, live in a minute.

Bigger changes (new page, new section, redesign)? Come back to this workspace and ask Claude, then re-download and re-upload the changed files.
